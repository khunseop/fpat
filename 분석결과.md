# CLI 구현 및 연동 분석 결과

본 문서는 `firewall_module`과 `policy_deletion_processor` 모듈의 CLI 구현 방식과 두 모듈을 함께 사용할 때 발생하는 불편함의 원인을 분석한 결과입니다.

---

## 1. 모듈별 CLI 구현 현황

### 1.1. firewall_module (`fpat/firewall_module/cli.py`)
*   **구현 방식**: 표준 `argparse` 기반의 CLI 도구.
*   **특징**:
    *   자동화에 최적화된 설계 (인자 주입, 환경 변수 활용 가능).
    *   명확한 출력 경로(`--output`) 지정 방식.
    *   단일 목적(데이터 추출 및 엑셀 저장)에 충실하며 사용법이 직관적임.

### 1.2. policy_deletion_processor (`fpat/policy_deletion_processor/`)
이 모듈은 두 가지 진입점을 가지고 있어 구조적 파편화가 존재합니다.

*   **진입점 A: `main.py` (대화형 메뉴)**
    *   **방식**: 번호를 입력받아 작업을 선택하는 메뉴 방식 (`input()`, `print()` 위주).
    *   **불편함**: 파일 선택 시 현재 디렉토리를 스캔하여 번호로 선택하게 함으로써, 자동화된 파이프라인(Piping) 구성을 불가능하게 만듦.
    *   **기능 범위**: 14가지의 상세 작업을 모두 포함함.

*   **진입점 B: `cli.py` (명령줄 인자)**
    *   **방식**: `argparse`를 사용한 비대화형 실행 시도.
    *   **불편함**: `main.py`에서 제공하는 14개 작업 중 3~4개(Parsing, Extracting, Usage 추가)만 구현되어 있음. `run_all` 기능 또한 전체 프로세스가 아닌 2단계만 수행하여 실질적인 자동화 도구로서의 가치가 떨어짐.

---

## 2. 두 모듈 연동 시 불편함의 근본 원인

두 모듈을 이어서 사용해야 하는 사용자 입장에서 느끼는 "불편함"은 다음과 같은 설계 불일치에서 기인합니다.

### 2.1. 자동화 vs. 대화형의 충돌
*   `firewall_module`은 "명령어 한 줄"로 결과물을 만들어내지만, 이를 처리하는 `policy_deletion_processor`는 "메뉴 번호 선택"과 "파일 번호 선택"이라는 사용자 개입을 강제합니다.
*   이로 인해 배시 스크립트(Bash Script) 등을 활용한 일괄 처리가 매우 어렵습니다.

### 2.2. 불완전한 CLI 래퍼
*   `policy_deletion_processor/cli.py`가 존재하지만, 실제 핵심 로직(14개 작업)을 모두 지원하지 못해 결국 다시 `main.py`로 돌아가야 하는 상황이 발생합니다.

### 2.3. 일관성 없는 파일 핸들링
*   `firewall_module`은 명시적인 파일 경로를 사용하지만, `policy_deletion_processor`는 내부적인 버전 관리 로직(`_v1`, `_v2`, `_vf` 접미사 자동 추가)과 대화형 파일 피커에 의존합니다.
*   추출된 결과 파일명이 프로세서의 버전 규칙과 맞지 않을 경우 파일 선택 과정에서 혼선이 생길 수 있습니다.

### 2.4. 통합 엔트리 포인트의 부재
*   "방화벽 정책 추출 -> 신청 정보 매칭 -> 사용현황 분석 -> 삭제 분류"로 이어지는 전체 워크플로우를 관장하는 통합 CLI 도구가 없어, 사용자가 각 단계별로 파일명을 확인하며 수동으로 명령어를 실행해야 합니다.

## 3. 컨피그(설정) 관리 분석 결과

현재 프로젝트의 설정 관리 방식은 다음과 같은 이유로 유지보수와 사용성이 떨어집니다.

### 3.1. 소스 코드에 종속된 설정 파일 경로
*   `ConfigManager`가 `config.json`을 찾을 때 `__file__`(현재 파이썬 파일 위치)을 기준으로 경로를 탐색합니다.
*   이로 인해 라이브러리를 설치하여 사용할 때 설정 파일을 찾지 못하거나, 사용자가 원하는 위치의 설정 파일을 불러오기가 매우 어렵습니다.

### 3.2. 모듈별 파편화된 설정 방식
*   **`policy_deletion_processor`**: `config.json` 파일에 의존하며, 파일이 없으면 정상 동작이 어렵습니다.
*   **`firewall_module`**: 벤더별 요구사항(`REQUIRED_PARAMS`) 등이 클래스 내부에 하드코딩되어 있어, 새로운 벤더 추가나 설정 변경 시 코드를 직접 수정해야 합니다.
*   **통합 설정 부재**: "미사용 정책 기준 일수(90일)"와 같은 핵심 비즈니스 로직 설정이 여러 모듈에 흩어져 있거나 중복 정의되어 있어 일관된 관리가 불가능합니다.

### 3.3. 환경별 설정 대응 미흡
*   개발, 테스트, 운영 환경에 따라 달라져야 할 설정(API 타임아웃, 로그 레벨, 파일명 규칙 등)을 환경 변수나 외부 주입 방식으로 처리하는 로직이 부족합니다.

---

## 4. 요약 및 제안 방향

현재 구현은 **`firewall_module`은 도구(Tool)**로서 잘 설계되어 있으나, **`policy_deletion_processor`는 개별 스크립트(Script)**들의 모음 수준에 머물러 있습니다. 또한 전체적으로 **설정 시스템의 유연성**이 부족합니다.

**개선 제안 (수정 전 방향성):**
1.  **CLI 통합**: `policy_deletion_processor/cli.py`를 확장하여 `main.py`의 14가지 기능을 인자로 모두 수용하도록 통합.
2.  **유연한 파일 선택**: `FileManager`의 대화형 파일 선택 로직을 비대화형(명시적 경로 인자)으로도 동작할 수 있게 개선.
3.  **설정 시스템 개편**: 
    *   중앙 집중형 설정 관리 도입 (예: `.env` 또는 프로젝트 루트의 `fpat.yaml`).
    *   설정 탐색 순서 도입 (명시적 인자 > 환경 변수 > 현재 디렉토리 설정 > 기본값).
4.  **통합 엔트리 포인트**: 두 모듈을 관장하는 최상위 워크플로우 스크립트(예: `fpat cli`) 도입을 통한 사용자 경험 통일.
